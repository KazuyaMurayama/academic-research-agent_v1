# Timeout Prevention — Claude Code 実行ガイド

最終更新: 2026-04-30（v2: 大JSON Read / エージェント分割 / 中間保存 / 並列化を追記）

---

## 根本原因

1レスポンスあたりの出力トークン上限（約8,192 tokens）が制約。
ファイル書き込み系API（push_files / create_or_update_file）はファイル全文をパラメータ文字列として出力するため、ファイルが大きいほどタイムアウトリスクが高まる。

| ファイル行数 | トークン概算 | 判定 |
|---|---|---|
| 200行以下 | ~5,000 | 安全 |
| 200〜280行 | ~6,500 | 要注意 |
| 280〜365行 | ~8,000 | 危険 |
| 365行超 | 8,000超 | タイムアウト |
| 625行（章結合等） | ~13,000 | 根本的に不可能 |

### 入力側のタイムアウト経路（v2追加）

出力トークンだけでなく **入力トークンの急膨張** もタイムアウト要因となる。
大きなJSON/データファイルをReadすると入力が一気に膨らみ、tool_useラウンドトリップ間のstream idle時間が長くなり、Stream idle timeoutで切断される（実測: 1.46MB JSONで234秒タイムアウト）。

| 入力ファイルサイズ | 挙動 |
|---|---|
| 〜100KB | 安全 |
| 100〜500KB | 要注意（必要部分のみ抽出を検討） |
| 500KB超 | **Read禁止**。Bashで事前抽出してインライン埋め込み |
| 1MB超（例: 1.46MB JSON） | 単独Readで確実にStream idle timeout |

## コア原則（5つ）

1. **Local-first, push-last:** ファイル生成はWrite/Editでローカル完結し、最後にBashでgit add && commit && pushする。push_filesは初期構築時のみ使用。
2. **1 commit = 1〜3 files:** push_filesのfiles配列は最大3要素。長文MDとメタ更新は必ず分離。
3. **長文は分割生成:** 1ターンあたり1,500トークン程度を目安に出力を区切り、セクション単位でコミットを挟む。
4. **読み込みは並列、書き込みは直列:** get_file_contentsは複数ファイルを1メッセージで並列実行可。create_or_update_fileは逐次実行（並列は422 Conflictの原因）。
5. **再生成しない:** 一度取得したファイル内容はコンテキストに保持されるため、結合時は再生成せず読み取った文字列をそのまま連結する。

## エージェント単位の負荷分割（v2）

単一エージェントへタスクを集約しすぎるとtool_useラウンドトリップが累積し、stream idle区間が増えてタイムアウトする。

### 1エージェントあたりの上限目安

| 項目 | 上限 |
|---|---|
| Readするファイル数 | 3ファイル以下 |
| Readする合計サイズ | 500KB以下 |
| 生成テキスト量 | 3,000字以下（約1セクション分） |
| 更新ファイル数 | 3ファイル以下 |
| git操作 | 最終結合エージェントに集約する |

### アンチパターン（実測でタイムアウト）

「複数大ファイルRead + 8,000〜12,000字生成 + 複数ファイル更新 + git操作」を1エージェントに集約 → Stream idle timeout（234秒）。

### 推奨パターン

- 1エージェント = 1セクション執筆（max 3,000字 / max 3ファイルRead）
- レポート全体は「セクション執筆エージェント × N」+「結合エージェント × 1」で構成

## 大JSON/データファイルの取り扱い（v2）

### 原則: 500KB超はReadしない

```
NG: 1.46MBのJSONを直接 Read → Stream idle timeout
OK: Bashで jq / python3 / head 等を使って必要部分のみ抽出 → プロンプトにインライン埋め込み
```

### 抽出パターン例

```bash
# キーと件数だけ取得
python3 -c "import json; d=json.load(open('large.json')); print({k: len(v) for k,v in d.items()})"

# 上位10件だけ抽出
jq '.papers[:10]' large.json

# 集計値のみ抽出
jq '{total: .total_hits, tier1: (.tier1 | length)}' large.json
```

## 中間保存プロトコル（v2）

### partial response receivedによる全消失リスク

中間保存を持たない設計で全文を1エージェントに生成させると、partial response received時に生成物が0バイトで失われる（実測で発生）。

### 必須プロトコル

1. セクション単位で **`outputs/` にローカル保存**（`outputs/sec_01.md`, `outputs/sec_02.md`, ...）
2. 各エージェントは担当セクションのみを書き出す
3. 全セクション完了後、別エージェントが `cat outputs/sec_*.md` で結合し `reports/` に最終保存

### 保存先のルール

| 種別 | 保存先 | git管理 |
|---|---|---|
| 中間ファイル（sec_01.md 等） | `outputs/` | gitignore済み（コミット不要） |
| 最終成果物 | `reports/` | コミット対象 |

## 並列エージェント実行（v2）

### 実測効果: 3〜4倍速

逐次実行で30分かかるレポート生成が、並列化で7〜10分まで短縮（Batch1: 4並列 → Batch2: 4並列 → Batch3: 2並列）。

### 並列化の安全条件

- 各エージェントが読み込むファイルが排他的であること
- 各エージェントが書き出す中間ファイルが排他的であること（番号で分離）
- git操作は並列禁止（最終エージェントに集約）

### バッチ設計テンプレート

```
Batch1（並列）: sec_01〜sec_04 を同時起動（各エージェントは異なるchunkを読む）
Batch2（並列）: sec_05〜sec_08 を同時起動
Batch3（並列）: sec_09〜sec_10 を同時起動
最終:  結合エージェントが cat outputs/sec_*.md → reports/ にWrite → commit
```

## ファイルサイズ別の推奨手法

### 200行以内
create_or_update_fileで1回pushして完結。

### 200〜280行
create_or_update_fileを使い、push_filesは避ける。

### 280行超（2分割パターン）

```
タスク1: _tmp/file_a.md（前半 ~140行）をpush
タスク2: _tmp/file_b.md（後半 ~140行）をpush
タスク3: 両方を読み込み→本体ファイルとして結合push
タスク4: _tmp/ 以下を削除してコミット
```

## タイムアウト発生時のフォールバック

| 症状 | 対処 |
|------|------|
| push_filesが大ペイロードでhang | Write→Bash pushに即切替 |
| MCP切断の疑い | 軽量readでプロービング（SHA取得確認） |
| 3回連続失敗 | チャット内にファイル内容を提示し、ユーザーに手動保存を依頼 |
| WebSearch長文が蓄積 | サブエージェントに分離してcontext隔離 |
| アシスタント出力が長大化しそう | セクション単位でコミットし、新ターンで再開 |

フォールバック手順（MCP障害時）:
1. 軽量readでプロービング
2. 成功したら送信サイズを半分以下にして再試行
3. 3回連続失敗ならユーザーに手動保存を依頼

## セッション中断・リカバリ

タイムアウトはシステム強制終了のため、Claude自身がリアルタイムに検出することは不可。
以下のプロトコルで途中停止リスクを最小化する。

**長時間タスク中の義務:**
- フェーズ完了ごとにprogress.mdを更新してコミット
- progress.mdに記録する内容: 完了済みフェーズ（[x]）とcommit SHA、次に実行すべき手順、再開に必要な状態メモ
- 1タスク完了 → commit → 次タスク のサイクルを徹底する

**セッション起動時の手順:**
1. progress.mdを読む
2. 「進行中タスクなし」以外の内容があれば続きから再開する
3. タスク完了後はprogress.mdを「進行中タスクなし」にリセットしてコミット
4. ユーザーは `claude --resume` で直前セッションのコンテキストを復元可能

## 失敗パターン一覧

| パターン | 理由 |
|---|---|
| 複数章を1ファイルに結合 | 行数・トークン超過で根本的に不可能 |
| append操作をAPI経由で実施 | GitHub APIにappendは存在しない |
| 書き込みを並列実行 | 422 Conflictエラーが発生 |
| push_filesで大ファイルを送る | create_or_update_fileよりオーバーヘッドが大きい |
| 内容を再生成しながら結合 | トークン消費が2倍になる |
| WebSearch raw結果をそのまま保存 | session.jsonが肥大化しコンテキストを圧迫 |
| ブランチを都度作成する | HEADの競合・管理コストが増大 |
| 1MB超JSONを直接Read | 入力トークン急膨張でStream idle timeout（実測: 234秒） |
| 大Read+長文生成+git操作を1エージェントに集約 | tool_useラウンドトリップ累積でタイムアウト |
| 中間保存なしで全文を1ターン生成 | partial response receivedで全消失（0バイト） |
| 書き込みエージェントを並列起動 | 中間ファイル衝突・git競合 |
| 中間ファイルを reports/ に保存 | 不要なコミットが大量発生 |

## 作業開始前チェックリスト

- [ ] 作成・更新するファイルは200行以内か（超える場合は分割設計を先行させたか）
- [ ] 書き込みを直列実行しているか（並列書き込みになっていないか）
- [ ] push_filesのfiles配列は3要素以下か
- [ ] WebSearch結果を即座に要約圧縮しているか（rawのままsession.jsonに貼っていないか）
- [ ] ブランチを不要に作成していないか（main固定で上書き）
- [ ] progress.mdを確認して作業順を把握したか
- [ ] Readする入力ファイルは500KB以下か（超える場合はBashで事前抽出したか）
- [ ] 1エージェントの担当範囲は「3,000字生成 / 3ファイルRead / 3ファイル更新」以内か
- [ ] レポートはセクション単位で `outputs/` に中間保存する設計になっているか
- [ ] 並列起動するエージェント間で読み書きするファイルは排他的か
- [ ] 中間ファイルが `outputs/`（gitignore済み）・最終成果物のみ `reports/` か
