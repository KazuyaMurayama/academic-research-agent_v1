# タイムアウト防止・大ファイル生成ルール

長大なMarkdownレポート（600-900行 / 60,000-90,000文字）を Claude Code 上で安全に生成するための実践レシピ。新セッション開始時にまずこのファイルを読むこと。

---

## 🎯 このルールが適用される条件

以下のいずれかに該当する場合は、**必ず本ルールの推奨パターン（Option C）に従う**こと。

| 条件 | 閾値 |
|---|---|
| 推定ファイルサイズ | 30,000文字超 / 400行超 |
| 推定セクション数 | 6セクション超（介入別・薬剤別・トピック別等の並列セクション） |
| Markdownレポート | `reports/YYYY-MM-DD_*.md` で複数の介入・比較対象を扱う場合 |
| Writeツール1回の出力推定 | 出力本文が15,000文字を超えそうなとき |
| Edit対象 | old_string が長大（200行超）または重複しがち |

### ⚠️ 高リスクシグナル

- 「網羅的に」「介入ごとに」「N本以上の論文を引用して」と要求されている
- INDEX.mdやFILE_INDEX.mdなど複数メタファイルの同時更新が必要
- 1つのアシスタント応答内で `Write` → 大きな `Edit` を連続で行おうとしている
- 過去のセッションで同種タスクが途中で停止した履歴がある

---

## 🚀 推奨パターン: Option C（スケルトン先出し → バッチEdit置換）

### 全体フロー

```
[Phase 1] 計画（Opus）         → 構成・セクション数・プレースホルダー名を決定
[Phase 2] スケルトンWrite       → 5,000-8,000文字 → 即commit
[Phase 3] バッチEdit × N回      → 4-6セクション/バッチ → 各バッチ後 commit
[Phase 4] 共通セクション差し込み → 結論・参考文献等 → commit
[Phase 5] メタデータ更新        → INDEX.md / FILE_INDEX.md / tasks.md → commit
[Phase 6] git push              → master に一括push
```

### Phase 1: 計画（Opus推奨）

実行前に以下を固める。**ここを飛ばすと後段で必ず手戻りが発生する**。

1. ファイル全体の見出し構造（H1/H2/H3）を箇条書きで列挙
2. 各セクションのおおよその文字数見積もり
3. **プレースホルダー名一覧**（後述の命名規則に従う）
4. バッチ分割（どのプレースホルダーをどのバッチで埋めるか）
5. コミットメッセージの雛形を5-7本ぶん用意

### Phase 2: スケルトンWrite

スケルトンには以下を含める:

- 全見出し（H1/H2/H3）
- 各セクションの**ユニークなプレースホルダー1行**（HTMLコメント形式）
- 既に書ける軽量セクション（メタ情報ヘッダ、目的、方法概要など 2,000文字以内）
- 目次（必要なら）

スケルトン例:

```markdown
# 〇〇に関する系統的レビュー

**作成日**: 2026-04-30
**対象論文数**: 42本

## 1. エグゼクティブサマリー
<!-- S1_EXECUTIVE_SUMMARY_PLACEHOLDER -->

## 2. 介入別エビデンス
### 2.1 有酸素運動
<!-- I01_AEROBIC_EXERCISE_PLACEHOLDER -->

### 2.2 禁煙
<!-- I02_SMOKING_CESSATION_PLACEHOLDER -->

## 3. 結論
<!-- S3_CONCLUSION_PLACEHOLDER -->

## 4. 参考文献
<!-- S4_REFERENCES_PLACEHOLDER -->
```

**Writeが終わったら即commit**（コミットしないとここで停止した瞬間に成果ゼロになる）。

```bash
git add reports/2026-04-30_xxx.md
git commit -m "feat: 〇〇レビューのスケルトン作成"
```

### Phase 3: バッチEdit × N回

1バッチ = 4-6セクション分のプレースホルダーをまとめて置換する。

- 1回の Edit ツール呼び出しで1プレースホルダーを置換
- 4-6個の Edit を**1つのアシスタント応答内**で並列実行（独立操作なので並列化可）
- バッチが終わったら**毎回commit**してからユーザー応答を返す

バッチ後のチェックコマンド:

```bash
grep -c "PLACEHOLDER" reports/2026-04-30_xxx.md   # 残プレースホルダー数
wc -l reports/2026-04-30_xxx.md                   # 行数進捗
```

### Phase 4-5: 仕上げ

- 結論・参考文献など他セクションに依存するブロックは最後に Edit
- `reports/INDEX.md` / `FILE_INDEX.md` / `tasks.md` の更新は別コミットに分離
- すべてのプレースホルダーが消えたことを `grep -c "PLACEHOLDER"` で確認 → 0なら完成

### Phase 6: push

```bash
git push origin master
```

## 🏷️ プレースホルダー命名規則

### 必須要件

1. **ファイル全体でユニーク**（Editのold_string重複エラーを回避するため）
2. **HTMLコメント形式** `<!-- ... -->`（最終出力でレンダリングされても見えない）
3. **大文字スネークケース + 末尾 `_PLACEHOLDER`**
4. **プレフィックスで分類**（セクション種別が一目で分かる）

### プレフィックス規則

| プレフィックス | 用途 | 例 |
|---|---|---|
| `S<番号>_` | 単発セクション（Summary, Conclusion等） | `S1_EXECUTIVE_SUMMARY_PLACEHOLDER` |
| `I<2桁番号>_` | 介入別（Intervention） | `I01_AEROBIC_EXERCISE_PLACEHOLDER` |
| `D<2桁番号>_` | 薬剤別（Drug） | `D03_METFORMIN_PLACEHOLDER` |
| `T<2桁番号>_` | トピック別（Topic） | `T05_BIOMARKER_PLACEHOLDER` |
| `M<番号>_` | メタ情報（Method, Limitation等） | `M2_LIMITATIONS_PLACEHOLDER` |
| `R<番号>_` | 参考文献ブロック | `R1_REFERENCES_PLACEHOLDER` |

### 良い例 / 悪い例

✅ 良い: `<!-- I07_INTERMITTENT_FASTING_PLACEHOLDER -->`
✅ 良い: `<!-- S2_METHODS_PLACEHOLDER -->`

❌ 悪い: `<!-- TODO -->`（ユニークでない）
❌ 悪い: `[INSERT HERE]`（HTMLコメントでないので残骸が表示される）
❌ 悪い: `<!-- PLACEHOLDER1 -->`（番号だけで意味不明、後段で混乱）

## 📏 バッチサイズ目安表

| フェーズ | 推奨サイズ | 限界値 | 備考 |
|---|---|---|---|
| スケルトンWrite | 5,000-8,000文字 | 12,000文字 | 一括Writeで完結する範囲 |
| 1プレースホルダーのEdit | 2,000-4,000文字 | 6,000文字 | new_stringのサイズ |
| 1バッチ（4-6セクション） | 12,000-20,000文字 | 25,000文字 | 1アシスタント応答の出力合計 |
| 1ファイル全体 | 60,000-90,000文字 | 制限なし（分割すれば可） | 30,000文字超は必ず分割 |

### セクション数別バッチ分割の目安

| 総セクション数 | バッチ数 | 1バッチあたり |
|---|---|---|
| 〜6 | 1-2 | 全部入れて可 |
| 7-12 | 2-3 | 4-5セクション |
| 13-20 | 3-4 | 4-5セクション |
| 21+ | 5以上 | 4セクション固定 |

### ❌ 一括Write禁止ライン

**30,000文字 / 400行を超える単一Writeは禁止**。超えそうな場合は必ずスケルトン+バッチEditに切り替える。

## 💾 git commit タイミングの原則

### 鉄則: Write/Edit → 即commit

**コミットされていない変更は存在しないものとして扱う**。中断・タイムアウト・コンテキスト切れで即座に消える。

### コミット粒度

| タイミング | コミットメッセージ例 |
|---|---|
| スケルトン作成直後 | `feat: 〇〇レビューのスケルトン作成` |
| バッチN完了直後 | `feat: 〇〇レビュー バッチ1（介入1-5）執筆` |
| 共通セクション完了 | `feat: 〇〇レビュー 結論・参考文献追加` |
| メタデータ更新 | `docs: INDEX.md / FILE_INDEX.md 更新` |

### コミット前チェック

```bash
git status                                       # 想定外のファイルが含まれていないか
grep -c "PLACEHOLDER" reports/対象ファイル.md   # 進捗確認
git diff --stat                                  # 変更行数の妥当性確認
```

### pushのタイミング

- バッチごとに push してもよい（推奨: 中断リスクが大きい長大タスク）
- 最低限、全バッチ完了後に必ず `git push origin master`

## 🛠️ リカバリー手順（中断時）

セッションが途中で停止した場合は、新セッションで以下を順番に実行する。

### Step 1: 状態確認

```bash
git log --oneline -10                            # 直近のコミット確認
git status                                       # 未コミット変更があるか
ls -la reports/                                  # 対象ファイルの存在確認
```

### Step 2: 進捗特定

```bash
grep -n "PLACEHOLDER" reports/対象ファイル.md   # 残プレースホルダー一覧（行番号付き）
wc -l reports/対象ファイル.md                   # 現在の行数
```

`grep -n` の出力 = 未処理セクションのリスト。これを見て続きを再開する。

### Step 3: 再開計画

1. 残プレースホルダーをセクション種別ごとにグルーピング
2. 4-6セクション/バッチで再分割
3. `tasks.md` に計画が残っているか確認 → 残っていればそれに従う、なければ再計画

### Step 4: 再開実行

- バッチEditを順次実行
- **各バッチ後の即commit鉄則は変わらず**

## ⚠️ よくある失敗パターンと対策

### 失敗1: 一括Writeでタイムアウト

**症状**: 80,000文字のレポートを1回の Write で生成しようとして応答が返らない / コンテキスト圧迫で停止

**対策**: 30,000文字超は問答無用でスケルトン+バッチEdit。「もう少しで終わるから1回でやろう」は禁物。

### 失敗2: Editのold_string重複エラー

**症状**: `Found N matches of the string to replace` エラーで Edit が失敗

**対策**:
- プレースホルダーで old_string のユニーク性を**事前に**担保する
- 通常テキストを old_string にしない（同一フレーズが他所にある可能性が常にある）

### 失敗3: コミット忘れによる成果消失

**症状**: バッチを3回回したあと停止 → 新セッションで `git log` を見ると Phase 2 までしかコミットされていない

**対策**: バッチ完了 → commit を**1つの不可分な手順**として扱う。次のバッチに進む前にcommitを脳内ルール化。

### 失敗4: プレースホルダー命名衝突

**症状**: `<!-- PLACEHOLDER -->` のような汎用名を複数箇所に置き、Editでどれを置換すべきか分からなくなる

**対策**: 命名規則（プレフィックス + 番号 + 内容）を厳守。スケルトン作成前にプレースホルダー一覧を必ず列挙。

### 失敗5: コンテキスト窓圧迫

**症状**: バッチを重ねるうちにアシスタント応答が肥大化し、後半のEditで応答が乱れる

**対策**:
- Edit の new_string に直接コンテンツを渡し、アシスタントメッセージで全文プレビューしない
- バッチごとに短い完了報告のみ出力する

### 失敗6: メタデータ更新忘れ

**症状**: レポート本体は完成したが `reports/INDEX.md` / `FILE_INDEX.md` / `tasks.md` の更新が漏れる

**対策**: 最終commit前に以下を必ず確認:

```bash
grep "対象ファイル名" reports/INDEX.md     # 登録されているか
grep "対象ファイル名" FILE_INDEX.md         # 登録されているか
grep "対象タスク" tasks.md                  # ステータスが完了になっているか
```

### 失敗7: モデル選択ミス

**症状**: 全工程をOpusで回してコスト超過 / 全工程をSonnetで回して計画が雑になる

**対策**: Phase 1（計画・プレースホルダー設計）= **Opus**、Phase 2-5（実行・Edit連打）= **Sonnet**。詳細は `.claude/rules/model-selection.md` 参照。

## 📋 開始前チェックリスト

新規大ファイル生成タスクを開始する前に確認:

- [ ] 推定文字数を見積もった（30,000文字超ならOption C必須）
- [ ] セクション構成を箇条書きで列挙した
- [ ] プレースホルダー名を全列挙し、ユニーク性を確認した
- [ ] バッチ分割（4-6セクション/バッチ）を決めた
- [ ] `git branch` で master 上にいることを確認した

## 📋 完了時チェックリスト

- [ ] `grep -c "PLACEHOLDER"` の結果が 0
- [ ] `wc -l` で想定行数に達している
- [ ] `reports/INDEX.md` 更新済み
- [ ] `FILE_INDEX.md` 更新済み
- [ ] `tasks.md` のステータスを「完了」に更新済み
- [ ] 全バッチがコミット済み（`git log --oneline` で確認）
- [ ] `git push origin master` 実行済み
