# 成果物・報告ルール

## 📋 回答冒頭ルール（毎回必須）

1. **プロンプト要約を冒頭に明記**してから本題に入る
2. 論理的・客観的・事実ベースで回答する（おべっか・楽観コメント禁止）
3. 意外な点・反直感的な情報は強調する（太字・⚠️等を活用）
4. ユーザーに判断を求めるときは選択肢＋推奨案を提示
5. できるだけClaude Code側でタスク実行し、ユーザー側タスクを最小化

## 📦 成果物ファイルの報告

- **GitHubハイパーリンクで必ず報告**（URL直貼り禁止）
- 形式: `[📄 ファイル名](https://github.com/KazuyaMurayama/academic-research-agent_v1/blob/master/パス/ファイル名)`
- ワンクリックで直接開けるリンクにすること
- ブランチは常に `master`（セッション跨ぎで確実にアクセス可能）

## 📄 レポートファイルのルール

- 保存先: `reports/YYYY-MM-DD_{英語スラグ}.md`
- 毎回 `reports/INDEX.md` を更新する（IDは R001〜連番）
- `outputs/` はgitignore済みのため `reports/` への保存が必須

## 🎯 レポート構成テンプレート（実践重視・毎回必須）

> **大前提**: レポートは「研究のため」ではなく **「ユーザーが実践して成果を出すため」** に書く。品質の定義は次の3つ：**①いかに早く読めるか ②いかに早く効果的なものを選べるか ③いかに簡単・スムーズに実行できるか**。学術的網羅性より、読者の行動・成果を優先する。

### 必須のセクション順序

1. **エグゼクティブサマリー** — 結論を先出し（目的別の推奨と反直感的な警告）
2. **目的・スコープ・読み方** — 短く。評価軸の定義のみ
3. **スコアリング比較テーブル（必ず序盤・「2」の直後）** — 一覧で素早く選べるように最前に置く
4. **目的別クイック選択／推奨プロトコル** — 読者の状況別にどれを選ぶか
5. **施策別 実践ガイド（A・Bランク、またはトップ5〜7のみ）** — 各施策に「具体的なやり方」を必ず入れる
6. （背景理論・科学的基盤があれば **後方**に。読みたい人だけ読む位置）
7. **注意点・安全性・免責**
8. **参考文献**

### 禁止・固定ルール

- ❌ **「エビデンスギャップ／今後の研究課題」セクションは作らない**（常に不要。読者の実践に寄与しない）。
- スコア表の列名は **「効果量／エビデンス強度／実行性」**（「実行」ではなく必ず **実行性**）。
- 施策別ガイドの詳細セクションは **A・Bランク（またはトップ5〜7）まで**。C・D／非推奨・研究段階のものは詳細を書かず、**1行サマリーの表**に集約する（重要な警告だけ残す）。

### 施策別 実践ガイドの各項目の書き方

| 項目 | ルール |
|---|---|
| 機序 | **1〜2文に絞る**。詳細な分子機構などは書かない |
| 効果量 | **代表的な2〜3個のみ**（最重要の数値）。羅列しない |
| エビデンス強度 | **1行**で重要な点のみ（CEBM Level＋推奨ガイドライン等） |
| ★具体的なやり方 | **最重要**。初心者が迷わず実行できるよう **ステップ・所要時間・具体例** を必ず盛り込む。自己実施できないもの（専門家必須）は「探し方・始め方・初回に何が起きるか」を書く |
| 注意点 | 簡潔に。安全上の警告を優先 |

### 読みやすさの必須要素

- 冒頭に **目次** と「お急ぎの方へ」の読む順ガイド
- 各施策に1行の「誰向け・何に効くか」サマリー
- 表・ステップ・チェックリストを活用し、長文段落を避ける

## 📝 tasks.mdの更新義務

- 新タスク開始時: tasks.md に追記
- タスク完了時: 即座にステータスを「完了」に更新してコミット
- 新セッション開始時: tasks.md を最初に参照

## 📁 FILE_INDEX.mdの更新義務

- 新ファイル追加時: FILE_INDEX.md に追記
- 削除・移動時: FILE_INDEX.md を更新

## 🔗 セッション継続困難時の処置

1. その旨をユーザーに伝える
2. 新セッション用引き継ぎプロンプトを回答に含める
3. 引き継ぎプロンプトには: 現在のタスク・完了済みステップ・次の手順・重要ファイルパスを明記
